free5GC ULCL on Kubernetes

Complete implementation guide: from bare-metal kubeadm cluster to verified ULCL traffic steering between PSA-UPFs with 8 UEs.
MARE Project · SNS JU Grant 101191436 · NCSR Demokritos · June 2026

Overview

This guide documents the full process of deploying a custom-patched free5GC v3.3.0 ULCL (Uplink Classifier) 5G core network on a 4-node kubeadm Kubernetes cluster, achieving bidirectional data-plane connectivity for 8 UEs, and implementing dynamic flow steering between two PSA-UPFs (anchor UPFs). The deployment replicates a working bare-metal testbed in a Kubernetes environment for the MARE project's autonomous self-healing research.

The work involved building 8 custom-patched NF Docker images from bare-metal binaries, debugging the full authentication chain, upgrading the gtp5g kernel module, resolving macvlan multi-route issues, and patching the SMF Go source code to enable uplink traffic steering. The result is a fully functional ULCL testbed where traffic can be dynamically steered between anchor UPFs via a REST API.

Phase 2: Custom Patched NF Images

Why Custom Images Are Required

The stock free5GC v3.3.0 images from the Helm chart have multiple bugs that prevent UE registration and PDU session establishment. The bare-metal testbed runs patched binaries that fix:

  • AMF: NGAP decode bypass, AUSF _links format fix, NRF discovery query fixes
  • SMF: BPManager FindULCL nil pointer guard, SendPfcpSessionDeletion nil check, charging URR bounds check, custom steering endpoint
  • AUSF/UDM/UDR: API version v2 compatibility (nudr-dr)
  • UPF: gtp5g 0.9.14 compatibility

Building Images with Buildah

Strategy: copy pre-built patched binaries from bare-metal (.137) and package them in minimal ubuntu:20.04 containers using buildah on .91.

Key difference: Stock schema uses permanentKey/opcKey; patched schema uses encPermanentKey/encOpcKey and sequenceNumber as an object with sqn + lastIndexes.
Critical — policy data format: The patched NFs expect a flat MongoDB structure (matching bare-metal), NOT the nested format from the free5GC documentation or webUI. Specifically:

policyData.ues.amData: must be {ueId, subscCats: [...]} — NOT {ueId, uesAmPolicyData: {subscCats: [...]}}

policyData.ues.smData: must be {ueId, smPolicySnssaiData: {...}} — NOT {ueId, smPolicyData: {smPolicySnssaiData: {...}}}

Using the wrong format causes the patched PCF to panic with nil pointer dereference on every SM policy query, blocking all PDU session establishment. The cold-start.sh script uses the correct bare-metal format.

Uplink: UE → DN

After gtp5g upgrade and UPF pod restart, uplink path was verified step by step:

Fix 1: AnchorUPF1 N6 egress routing

With multiple macvlan interfaces on the same /24, the default route sent N6 traffic via the wrong interface:

# Inside AnchorUPF1 pod:
ip route replace 10.160.101.145/32 dev n6
# This ensures DN-bound traffic exits via n6, not n9

Fix 2: DN server return route

The DN server (.145) is shared between bare-metal and K8s. Return routes must point at K8s anchors:

# On DN server (.145):
sudo ip route replace 10.61.0.0/16 via 10.160.101.123  # AnchorUPF1 N6
sudo ip route replace 10.62.0.0/16 via 10.160.101.133  # AnchorUPF2 N6

Fix 3: NAT removal

Bare-metal uses routed return (no MASQUERADE). K8s conntrack was 0, so NAT couldn't un-NAT replies:

# Inside each UPF pod:
iptables -t nat -F  # flush NAT rules

Phase 5: Flow Steering

UE Routing Configuration

The uerouting.yaml configmap defines the ULCL topology and specificPaths. For the steer to work, AnchorUPF2 must have a pool, and there must be a specificPath with dest matching the traffic destination:

info:
  version: 1.0.7
  description: Routing information for UE

ueRoutingInfo:
  UE1:
    members:
    - imsi-208930000000001
    - imsi-208930000000002
    - imsi-208930000000003
    - imsi-208930000000004
    - imsi-208930000000005
    - imsi-208930000000006
    - imsi-208930000000007
    - imsi-208930000000008
    topology:
      - A: gNB1
        B: BranchingUPF
      - A: BranchingUPF
        B: AnchorUPF1
    specificPath:
      - dest: 10.160.101.145/16
        path: [BranchingUPF, AnchorUPF2]
      - dest: 10.160.101.145/16
        path: [BranchingUPF, AnchorUPF1]
      - dest: 8.8.8.8/32
        path: [BranchingUPF]
Key insight: AnchorUPF2 must have a pool (10.62.0.0/16) in smfcfg.yaml for the SMF to establish PSA2 sessions with proper PDRs. Without the pool, "No more valid PSA2 path found" appears and AnchorUPF2 gets 0 PDRs.

SMF Source Code Patch — Default FAR Steering

This is the critical fix that makes steering actually redirect traffic. The original HandleSteerTraffic in steering.go only modified SDF-filtered FARs on non-default paths (downlink direction). Uplink traffic (UE → DN) matched the default PDR and always went to AnchorUPF1 regardless of the steer.

Patch Add default-path uplink FAR modification to NFs/smf/internal/sbi/processor/steering.go:

// ── Insert BEFORE "if len(farList) == 0" ──

// Also redirect the DEFAULT-path uplink FAR on the I-UPF
if defaultPath.FirstDPNode != nil &&
    defaultPath.FirstDPNode.UpLinkTunnel != nil &&
    defaultPath.FirstDPNode.UpLinkTunnel.PDR != nil {
    defFAR := defaultPath.FirstDPNode.UpLinkTunnel.PDR.FAR
    if defFAR != nil && defFAR.ForwardingParameters != nil {
        existingTEID := uint32(0)
        if defFAR.ForwardingParameters.OuterHeaderCreation != nil {
            existingTEID = defFAR.ForwardingParameters.OuterHeaderCreation.Teid
        }
        defFAR.ForwardingParameters.OuterHeaderCreation = &pfcpType.OuterHeaderCreation{
            OuterHeaderCreationDescription: pfcpType.OuterHeaderCreationGtpUUdpIpv4,
            Ipv4Address:                    targetIP,
            Teid:                           existingTEID,
        }
        defFAR.State = smf_context.RULE_UPDATE
        pdrList = append(pdrList, defaultPath.FirstDPNode.UpLinkTunnel.PDR)
        farList = append(farList, defFAR)
        debugFARs += fmt.Sprintf("FARID=%d(default),UPF=%s;",
            defFAR.FARID,
            defaultPath.FirstDPNode.UPF.NodeID.ResolveNodeIdToIp().String())
        if iUPFNode == nil {
            iUPFNode = defaultPath.FirstDPNode
        }
    }
}

Also remove the early return on empty ulcfNodes (since the default-path FAR provides a valid FAR even without SDF nodes).

#!/bin/bash

Agent loop

The agent polls every ~4 seconds, running two parallel probes:

  1. Ping probe — SSH to I-UPF host, ping the AnchorUPF1 host to measure RTT and loss
  2. Flood probe — SSH to I-UPF host, tcpdump on ens18 for GTP-U packets, extract 11 features

Features are scaled and fed to the LSTM classifier (5 classes, CLASS_GTP_FLOOD=4). If GTP_FLOOD is detected for CONFIRM_SECS (4s), the agent steers the malicious UE SUPIs to AnchorUPF2.

SMF Rebuild Script (rebuild-smf.sh)

After any SMF source change (e.g., steering patch), this script rebuilds the image and deploys it. Critical because the K8s image cache aggressively holds stale layers — the script uses ctr images rm to force a fresh pull.

#!/bin/bash
set -e
REG=10.160.101.91:32000

echo "=== Copying patched binary ==="
cd ~/imgbuild
cp /tmp/smf-fixed bin/smf
ls -la bin/smf

echo "=== Building image ==="
c=$(sudo buildah --storage-driver vfs from ubuntu:20.04)
sudo buildah --storage-driver vfs run $c -- bash -c \
  "apt-get update && apt-get install -y libsctp1 iproute2 iptables ca-certificates && rm -rf /var/lib/apt/lists/*"
sudo buildah --storage-driver vfs run $c -- mkdir -p /free5gc/bin /free5gc/config /free5gc/cert
sudo buildah --storage-driver vfs copy $c bin/smf /free5gc/bin/smf
sudo buildah --storage-driver vfs copy $c cert/ /free5gc/cert/
sudo buildah --storage-driver vfs run $c -- chmod +x /free5gc/bin/smf
sudo buildah --storage-driver vfs config --workingdir /free5gc/bin $c
sudo buildah --storage-driver vfs commit $c $REG/free5gc-smf:patched
sudo buildah --storage-driver vfs push --tls-verify=false $REG/free5gc-smf:patched
sudo buildah --storage-driver vfs rm $c

echo "=== Clearing cache + restarting SMF ==="
sudo ctr -n k8s.io images rm $REG/free5gc-smf:patched 2>/dev/null
kubectl rollout restart deployment/free5gc-v1-free5gc-smf-smf -n free5gc
kubectl rollout status deployment/free5gc-v1-free5gc-smf-smf -n free5gc

echo "=== Restarting port-forward ==="
pkill -f "port-forward.*8000" 2>/dev/null; sleep 1
smf=$(kubectl get pod -n free5gc -o name | grep smf | grep -v ausf | head -1 | sed 's|pod/||')
kubectl port-forward -n free5gc $smf 8000:80 >/tmp/pf.log 2>&1 &
sleep 3

echo "=== Verifying binary ==="
kubectl exec -n free5gc $smf -c smf -- ls -la /free5gc/bin/smf
echo "MUST show 32442720 bytes."
Image cache trap: If the same tag (:patched) keeps pulling the old image, use a different tag (e.g., :v2) and kubectl set image to point the deployment at it. This forces K8s to pull a genuinely new image.

Operational Notes

  • kubectl only works on .91 (the control-plane node).
  • Port-forward dies on every SMF restart — always restart it: pkill -f "port-forward.*8000"; smf=$(kubectl get pod -n free5gc -o name | grep smf | grep -v ausf | head -1 | sed 's|pod/||'); kubectl port-forward -n free5gc $smf 8000:80 >/tmp/pf.log 2>&1 &
  • UE sessions exist only while nr-ue runs. Every SMF restart drops all sessions. Must relaunch UEs on .95 after any SMF change. Steering returns "SMContext not found" without live sessions.
  • Same-tag image updates need cache clearing: sudo ctr -n k8s.io images rm <img> on the node, then delete the pod.
  • Do NOT run helm upgrade — it reverts all configmaps, image overrides, and deployment annotations.
  • Runtime route fixes are lost on pod restart. For persistence, add routes to the UPF wrapper.sh configmap (e.g., AnchorUPF1's /32 route via n6, DN return routes).
  • gtp5g-tunnel (from libgtp5gnl) was built on .93 and copied into UPF pods for PDR/FAR inspection. Not baked into the UPF image — must re-copy after pod restarts.
  • TEIDs change on every UE relaunch. Always re-check TEIDs before running the flood script. Use the PDR dump command in the f_k8s.py docstring.
  • UE IPs increment if the SMF pool allocator remembers previous assignments. To reset to 10.61.0.1–8, restart the SMF first (clears the pool), then relaunch UEs.
  • Agent requires SSH key access from .91 to .92 (I-UPF host for tcpdump) and .94 (AnchorUPF2 host for traffic confirmation). Set up with ssh-copy-id.
  • steering.sh vs steering-k8s.sh — the bare-metal script uses 127.0.0.2:8000 (SMF runs locally on .137). The K8s script uses 127.0.0.1:8000 (port-forward on .91). Using the wrong one gives FAILED for all UEs.

Recovery Procedure

After extended operation, overnight restarts, or pod rescheduling, the testbed may need a full recovery. The symptoms and their fixes follow a predictable chain. This section provides both a quick full-restart recipe and a symptom-based troubleshooting guide.

Full Cold-Start Recovery (from scratch)

Run these steps in order on .91 unless otherwise noted. This handles every known failure mode.

Automated Recovery: cold-start.sh

The cold-start.sh script (on .91 at ~/cold-start.sh) performs all 11 recovery steps in sequence. Run it after any shutdown, crash, or overnight restart:

~/cold-start.sh

It handles: MongoDB probe fix → subscriber data insert (correct bare-metal format) → gtp5g reload on all 3 UPF nodes → PCF scale-up → NRF + all NF restarts → patched SMF verification → PFCP association check → runtime data-plane fixes → port-forward restart. After the script completes, launch gNB + UEs on .95.

Manual Recovery Steps

If you need to run individual steps or debug a specific issue, the manual procedure follows.

1 Fix MongoDB probes + reload gtp5g on UPF nodes

MongoDB 6.0 uses mongosh, not mongo. The Helm chart installs probes with the deprecated mongo CLI, causing liveness failures, pod restarts, and data loss. Fix permanently:

# Fix MongoDB probes (one-time, persists across restarts)
kubectl patch statefulset mongodb -n free5gc --type='json' -p='[
  {"op":"replace","path":"/spec/template/spec/containers/0/livenessProbe/exec/command",
   "value":["mongosh","--quiet","--disableImplicitSessions","--eval","db.adminCommand(\"ping\")"]},
  {"op":"replace","path":"/spec/template/spec/containers/0/readinessProbe/exec/command",
   "value":["bash","-ec","mongosh --quiet --disableImplicitSessions --eval \"db.hello().isWritablePrimary || db.hello().secondary\" | grep -q true"]}
]'

# Wait for MongoDB to restart with fixed probes
kubectl get pod -n free5gc -w | grep mongo
# Wait for 1/1 Running

If gtp5g is in a bad state (upfgtp RX=0, TX errors > 0), also reload it:

Critical — reload gtp5g on ALL THREE nodes (.92, .93, .94): After an overnight shutdown, gtp5g goes stale on every UPF node, not just the BranchingUPF. The correct order is: delete all UPF pods → wait for termination → reload gtp5g on all nodes → let new pods start on the fresh module. If you reload gtp5g while pods are already running, the pods hold stale handles and you'll see CreatePDR error: no such file or directory / upfgtp does not exist. The cold-start.sh script handles this ordering automatically.
